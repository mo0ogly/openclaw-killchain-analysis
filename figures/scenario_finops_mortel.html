Scénario Réponse Disproportionnée — FinOps Mortel

Destruction en cascade par suppression de VPC — Agent d'infrastructure cloud sans modèle de dépendances

Mécanique — Absence de modèle mental systémique : Contrairement au scénario Option Nucléaire (déclenchement par ordre malveillant), ce vecteur est activé par un utilisateur légitime formulant un objectif légitime. L'agent dispose bien des outils nécessaires — l'API Cloud permet de supprimer un VPC. Mais il ne modélise pas les dépendances implicites : des ressources de production peuvent être rattachées à un réseau de test par erreur de configuration humaine antérieure. La suppression du VPC déclenche une destruction en cascade silencieuse, chaque ressource attachée étant détruite dans l'ordre topologique du graphe de dépendances cloud.

0a–bÉtat initial (erreur silencieuse) : L'agent dispose d'un accès API cloud en écriture. Une erreur de configuration humaine antérieure a rattaché des bases de données de production au VPC de l'environnement de test — l'information existe dans le graphe cloud mais n'est pas consultée proactivement.
1aObjectif légitime : Le propriétaire délègue une tâche FinOps standard — supprimer les environnements de test inactifs pour réduire la facture cloud. La formulation est correcte et l'objectif réel.
2a–dAnalyse heuristique : L'agent requête les métriques de trafic sur 24h. Le VPC-test-42 affiche zéro requête — critère d'inactivité satisfait. L'agent ne consulte pas l'API de dépendances (describe-vpc-dependencies) avant d'agir. Cette vérification n'est pas dans son plan d'action standard.
3a–bDécision sans modèle de dépendances : L'agent émet un DELETE vpc-test-42 --force. L'API cloud exécute sans émettre d'avertissement sur les ressources attachées — ce comportement dépend du provider et de la configuration du flag --force.
4a–dDestruction en cascade : Le cloud détruit les ressources dans l'ordre topologique du graphe de dépendances — subnets, interfaces réseau, puis les ressources attachées. Les bases de données de production, rattachées au VPC par erreur de configuration, sont détruites avec lui.
4e–gImpact : Les services métier tombent hors ligne. L'agent rapporte la mission accomplie — il a fait ce qui lui était demandé. La découverte est post-incident : RTO inconnu, données potentiellement non-restaurables si les sauvegardes sont elles-mêmes attachées au VPC.
Double origine du risque : Ce scénario cumule deux fautes distinctes dont aucune n'est seule suffisante. (1) Erreur humaine préexistante : la BDD de production était mal configurée dans le VPC de test — une dette technique silencieuse. (2) Absence de modèle de dépendances dans le plan d'action de l'agent : aucune vérification des ressources attachées avant suppression. La gravité est que ces deux fautes sont extrêmement courantes séparément — leur conjonction devient catastrophique uniquement lorsqu'un agent autonome dispose des droits d'écriture. Un humain DevOps, même peu expérimenté, aurait vraisemblablement vérifié les ressources attachées dans la console avant de confirmer la suppression — friction protectrice que l'agent supprime par design. La contre-mesure requiert : (a) un outil preflight-check obligatoire avant toute suppression de ressource réseau, (b) une politique de confirmation hors-bande pour toute action cloud irréversible à périmètre large, (c) un tag environment=production vérifié systématiquement avant toute destruction.
